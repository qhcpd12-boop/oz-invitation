# 개발기획서 — 모바일 청첩장 서비스

## 1. 서비스 개요

### 1.1 한 줄 정의

사용자가 **요금제를 구매**하고 **결혼(예식) 정보를 입력**하면, 선택한 템플릿을 바탕으로 **모바일 청첩장(URL 또는 공유용 페이지)**을 생성·제공하는 웹 서비스다.

### 1.2 핵심 사용자 플로우

1. 비로그인 사용자: 랜딩에서 가치 인지 → (선택) 템플릿 미리보기 → **회원가입/로그인**
2. 로그인 사용자: **요금제 선택·결제(또는 무료 체험 정책이 있다면 그에 따른 진입)** → **STEP: 디자인 선택** → **STEP: 청첩장 정보 입력** → **(필요 시) 결제 완료 확인** → **청첩장 미리보기/공유 URL 발급**
3. 하객: 발급된 URL로 모바일 최적화 청첩장 열람(RSVP·갤러리 등은 요금제·기획 범위에 따름)

### 1.3 디자인·기획 기준 문서

| 구분 | 경로·URL |
|------|-----------|
| 화면 구조·카피·섹션 단위 IA | 저장소 내 `화면설계서.md` |
| 시각 시안(Figma) | [Figma — node `1:1246` (Cover)](https://www.figma.com/design/CUPgvY0YOqis5pZB7hufGP/Full-website-design-Templates-for-Cyprus-University--Community-?node-id=1-1246&t=aRJ8KxiXG167vLNV-1) |

구현 시 **레이아웃·타이포·여백·컴포넌트 배치**는 위 Figma와 `화면설계서.md`의 섹션 구조(Nav, Hero, Stats, Features, Testimonials, Footer, CTA, 회원가입, 요금제, STEP 1~3)를 우선 준수한다. 반응형은 데스크톱 기준(설계서상 1920×1200 뷰)을 출발점으로 **MUI 브레이크포인트**로 재배치한다.

## 2. 기술 스택

| 영역 | 기술 | 비고 |
|------|------|------|
| UI 프레임워크 | **React**(권장: Vite + TypeScript) | 빌드 속도·Vercel 친화적 |
| 컴포넌트·레이아웃 | **MUI(Material UI) v5/v6** | 테마로 브랜드 컬러·타이포 통일 |
| 인증 | **Firebase Authentication** | 이메일/비밀번호, 이후 소셜(카카오 등)은 별도 검토 |
| 데이터베이스 | **Cloud Firestore** | 청첩장 문서·사용자·주문 메타 등 |
| 호스팅 | **Vercel** | 정적 프론트 + 서버리스 API Routes(선택) |
| 결제 | **본 문서 범위에서 별도 PG** | Stripe/토스페이먼츠 등 + 웹훅 검증 권장(아래 6절) |

Firebase **Hosting만** 쓰지 않고 Vercel에 올리는 경우에도, 클라이언트에서 Firebase SDK로 Auth·Firestore에 직접 연결하는 패턴이 일반적이다.

## 3. 애플리케이션 구조(권장)

### 3.1 라우팅 개요

`화면설계서.md`의 IA를 **페이지 단위**로 나누는 것을 권장한다.

| 경로(예시) | 화면 | Figma 대응 |
|------------|------|-------------|
| `/` | 랜딩(스크롤 섹션: Nav~CTA) | Hero, Stats, Features, Testimonials, Footer, CTA |
| `/signup`, `/login` | 회원가입·로그인 | 회원가입 패널·Nav의 로그인/무료 시작 |
| `/pricing` | 요금제·비교표·애드온 | 요금제 영역 |
| `/create` | 제작 위저드(서브라우트 또는 스) | STEP 1~3 |
| `/create/design` | 템플릿 선택 | STEP 1 |
| `/create/details` | 예식·커플 정보 입력 | STEP 2 |
| `/create/checkout` | 주문·결제 | STEP 3 |
| `/i/:slug` 또는 `/invite/:id` | **공개 청첩장 뷰**(하객용) | 디자인상 폰 목업의 “청첩장 보기”에 해당하는 실제 페이지 |

랜딩 한 페이지에 모든 블록을 넣되, **앵커 링크**(`#pricing`, `#features` 등)로 Nav와 연결하면 시안과 동작이 잘 맞는다.

### 3.2 MUI 적용 방침

- **ThemeProvider**로 primary/secondary, `typography.fontFamily`(Noto Sans KR 등) 설정해 시안과 근접하게 맞춘다.
- 섹션은 `Container` + `Stack`/`Grid2`로 정렬, 카드는 `Paper` 또는 `Card`로 **borderRadius 24** 등 설계서 수치를 토큰화한다.
- 폼은 `TextField`, `Checkbox`, `Button`, `Stepper`로 STEP 플로우를 구성한다.

### 3.3 상태·데이터

- **React Query(TanStack Query)** 또는 **Firestore onSnapshot**으로 청첩장 초안 동기화(선택).
- 위저드 중간 저장: `invitations/{id}` 문서에 `status: draft`, `step`, `formData` 등을 저장해 이탈 후 복귀 가능하게 한다.

## 4. Firebase 설계(초안)

### 4.1 Authentication

- **이메일/비밀번호**를 1차로 구현하고, 시안의 소셜 버튼은 UI만 두거나 **Custom Token** 연동 시점을 별도 마일스톤으로 둔다.
- 가입 직후 Firestore `users/{uid}` 프로필 문서 생성(Cloud Function 트리거 또는 클라이언트 1회성 쓰기 — 보안 규칙과 함께 설계).

### 4.2 Firestore 컬렉션(예시)

| 컬렉션 | 용도 | 주요 필드(예) |
|--------|------|----------------|
| `users/{uid}` | 공개 비민감 프로필 | `displayName`, `createdAt` |
| `users/{uid}/private/profile` | 민감정보 분리(또는 서버만 접근) | 정책에 따라 구조 조정 |
| `templates/{templateId}` | 선택 가능 템플릿 메타 | `name`, `thumbnailUrl`, `tier: light|standard|premium` |
| `invitations/{invitationId}` | 청첩장 본문 | `ownerId`, `templateId`, `wedding`, `venue`, `slug`, `publishedAt`, `status` |
| `orders/{orderId}` | 주문·결제 메타 | `userId`, `plan`, `amount`, `paymentProvider`, `paidAt` |

`invitationId`와 공개 URL의 `slug`는 유니크 인덱스·중복 검사 규칙을 둔다.

### 4.3 보안 규칙(원칙)

- `invitations`: **소유자만 쓰기**, `published`된 문서만 **누구나 읽기**(하객용) 또는 별도 `publicInvites/{slug}` 미러 컬렉션.
- `orders`: **본인만 읽기**, 쓰기는 **서버(Admin SDK)**만 허용하는 편이 안전하다(Vercel Serverless + Firebase Admin으로 결제 웹훅 후 주문 확정).

## 5. “구매 후 정보 입력 → 청첩장 완성” 구현 기준

1. **구매 완료** 시점에 `orders`를 확정하고, `invitations` 초안을 생성한다(`templateId`가 이미 정해져 있으면 함께 저장).
2. 사용자가 STEP 2 폼을 저장할 때마다 `invitations`의 하위 필드(신랑·신부, 예식 일시·장소, 인사말 등)를 갱신한다.
3. **완성** 조건(예: 필수 필드 검증 통과 + `publish` 클릭)에서 `status: published`, `slug` 확정, 공개 URL을 사용자에게 표시한다.
4. 공개 페이지(`/i/:slug`)는 MUI가 아닌 **가벼운 뷰**로 분리할지, 동일 스택으로 갈지는 성능·브랜딩에 따라 선택한다.

## 6. 결제·Vercel

- **결제 자체는 Firebase에 포함되지 않는다.** PG사 SDK 또는 Checkout Session + **서버에서만** 시크릿 키 사용(Vercel Environment Variables).
- 결제 성공 웹훅에서만 `orders`·`invitations` 권한을 갱신해 **클라이언트 조작으로 유료 기능이 열리지 않게** 한다.
- Vercel: `firebase` 클라이언트 설정은 `VITE_` 또는 `NEXT_PUBLIC_` 접두사 규칙에 맞게 노출하고, **Admin SDK 키는 절대 클라이언트에 넣지 않는다.**

## 7. 개발 단계(마일스톤 제안)

| 단계 | 내용 |
|------|------|
| M1 | Vite+React+TS, MUI 테마, 랜딩 `/` 시안 반영(정적 섹션) |
| M2 | Firebase Auth 연동, 회원가입/로그인 화면 |
| M3 | Firestore `invitations` 초안 CRUD, `/create` 스텝퍼 |
| M4 | 템플릿 목록 `templates` 연동(또는 초기에는 JSON 목업) |
| M5 | 공개 청첩장 `/i/:slug`, 모바일 퍼스트 레이아웃 |
| M6 | 결제 PG + 웹훅 + 주문 확정(운영 필수) |
